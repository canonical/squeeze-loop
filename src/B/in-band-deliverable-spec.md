# System Specification: The In-Band Deliverable Layer (Example B: Autonomous Refund Bot)

This specification outlines the structural, interface, and execution requirements for the code artifacts built inside the `implementer` and `exerciser` environments. These components occupy the *in-band* deliverable space, bounded from above by the dynamic **Authored Policy Spec (Upper Bound)** and from below by the **Headless REST API Runtime (Lower Bound)**.

To maintain compliance with physical isolation barriers (**C3**), these two execution bands operate under total context-blindness relative to each other. Their alignment is forced exclusively through deterministic interactions with the REST API platform during gate evaluation.

## 1. The Implementer’s Band: Prompt & Routing Architecture

The `implementer` account is responsible for producing the operational decision core of the bot. This code must process unstructured conversational inputs and dynamically execute downstream endpoint states based on the active policy.

* **File Target:** `/home/implementer/src/refund_bot_logic.py`
* 
**Context Environment:** This workspace contains the raw text of the active policy clauses (`spec.txt`) but is physically restricted from viewing any test scenario or validation script generated by the exerciser.



### Interface Requirements

The script must expose a programmatic interface or execution block that the orchestrator can call during multi-turn testing. It must accept a customer message alongside any existing session state metadata:

```python
def process_customer_turn(session_token: str, customer_message: str) -> dict:
    """
    Invoked by the sentinel on every chat turn.
    Interacts with the LLM prompt matrix and returns the next action.
    """
    pass

```

### Execution Constraints

1. **Indirect State Updates:** The bot logic must never write or update database records directly. It must execute all choices through the local REST API runtime.
2. **Explicit Payload Resolution:** To close an interaction, the implementer's logic must explicitly invoke the lower bound endpoint `POST /api/session/action` with one of the authorized parameters: `REIMBURSE`, `DENY`, or `ESCALATE`. Plain-text declarations inside the chat message stream are non-binding.

## 3. The Squeeze Connector: Multi-Turn Simulation Runner

The execution bridge is managed by the automated orchestrator component (`gate_sentinel.py`) under the `sentinel` user account. It serializes data flows between the two isolated agent scopes, enforcing the *squeeze* over the headless application environment.

For every scenario defined in the exerciser's matrix, the runner executes the following evaluation loop:

```python
import json
import requests
import sys

BASE_URL = "http://127.0.0.1:8000/api/session"

def run_in_band_squeeze():
    # Load the independent validation scenarios written by the Exerciser
    with open("/home/exerciser/scenarios/adversarial_matrix.json") as f:
        matrix = json.load(f)

    # Import the Implementer's processing logic dynamically
    sys.path.append("/home/implementer/src")
    import refund_bot_logic

    for scenario in matrix["scenarios"]:
        # Step 1: Initialize the session state on the Lower Bound App
        start_resp = requests.post(f"{BASE_URL}/start", json={
            "case_id": scenario["scenario_id"],
            "customer_id": scenario["customer_id"]
        })
        session_token = start_resp.json()["session_token"]

        # Step 2: Simulate the Multi-Turn Conversation
        for turn in scenario["turns"]:
            # Route customer message to the Implementer's logic block
            bot_output = refund_bot_logic.process_customer_turn(
                session_token=session_token, 
                customer_message=turn["message"]
            )
            
            # Post the bot's reaction back to the Lower Bound App conversation layer
            requests.post(f"{BASE_URL}/chat", json={
                "session_token": session_token,
                "message": bot_output.get("message", "")
            })

        # Step 3: Enforce Gate B (Verify Terminal Endpoint Commitment)
        # Fetch the actual action captured by the application's action lockpoint
        final_state = requests.get(f"{BASE_URL}/state/{session_token}").json()
        actual_action = final_state.get("committed_action")  # REIMBURSE, DENY, ESCALATE

        if actual_action != scenario["expected_terminal_action"]:
            print(f"GATE B CRASH: Scenario {scenario['scenario_id']} failed!")
            print(f"Expected final endpoint state: {scenario['expected_terminal_action']}")
            print(f"Actual implementer action committed: {actual_action}")
            sys.exit(1)

    print("GATE B SUCCESS: All in-band interactive paths align with policy rules.")

```

## 4. Automated Gate Coupling Focus

* 
**Gate B Clearance:** The runner validates that the implementer's prompt parameters resist conversational drift, capturing social engineering tracks or legal maneuvers and steering them cleanly into the target REST destination states specified by the exerciser.


* 
**Gate C Structural Check:** The sentinel extracts the `target_clauses` string arrays from the exerciser's json file. If any unique clause ID defined within the upper bound handbook (`refund_policy.md`) lacks a corresponding targeted testing scenario block here, the pipeline aborts for coverage deficiency.
