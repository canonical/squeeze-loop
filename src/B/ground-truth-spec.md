# System Specification: Ground Truth & Lower Bound Engine (Example B: Autonomous Refund Bot)

This specification defines the low-level source of truth (the executable lower bound) and environment architecture for **Archetype B (Authored Authority)**. Unlike transcription tasks with clear external reference documents, this system operates on a dynamically authored policy. Correctness is enforced by anchoring the multi-agent workflow against an unalterable, interactive local REST application and an immutable archive of historical interaction verdicts.

---

## 1. System Environment & Directory Architecture

The system executes within an isolated LXC container. It enforces physical context barriers among three distinct POSIX user accounts to eliminate epistemic data leakage and conversational contamination.

* `sentinel` (The Orchestrator and Gate Evaluator)
* `implementer` (The Prompt/Routing Agent workspace)
* `exerciser` (The Adversarial Test/Scenario Agent workspace)

### Directory Layout

```text

├── shared/                  # Immutable Low-Level Sources of Truth
│   ├── customer_history.db  # Step 1: Storage Plane (Read-Only SQLite)
│   └── archive_ledger/      # Step 2: Invariant Plane (Adjudicated Archive)
```

## 2. Low-Level Source of Truth Execution Planes

The executable lower bound consists of three localized layers that the agents cannot manipulate or rewrite.

### Step 1: Storage Plane (Frozen Account States)

* **Artifact:** `/opt/squeeze/shared/customer_history.db`
* **Mechanic:** A local SQLite database file containing static user runtime state information (e.g., account registration age, lifetime order count, past return velocity, fraud flags).
* **Enforcement:** This file is set to Unix file permissions `0444`. The runtime application accesses it strictly in read-only mode, blocking agents from artificially adjusting user histories to satisfy a flawed decision track.

### Step 2: Invariant Plane (The Adjudicated Archive)

* **Artifact:** `/opt/squeeze/shared/archive_ledger/`
* **Mechanic:** A historical regression baseline containing certified support conversations paired with immutable categorical badges: `[REIMBURSE]`, `[DENY]`, or `[ESCALATE]`.
* **Enforcement:** Every time model adjustments or prompt changes are checked in, the system sequentially fires the entire regression ledger against the runtime application. Any unauthorized shift in a historical outcome (a decision flip) trips **Gate B** and halts deployment, rendering regression an architectural impossibility.



### Step 3: Compute Plane (The Headless REST Interface App)

* **Artifact:** An isolated Python application running locally within the LXC container under a dedicated service user. It abstracts the environment constraints and downstream tools into deterministic endpoints.
* **Enforcement:** The application exposes a strict, non-conversational API boundary that the agents must call programmatically. Plain-text claims inside the chat window are ignored; only explicit REST calls alter state.

## 3. Ground Truth REST API Specification

The local application exposes the following interface to process client sessions. All transactions must interact through these fixed endpoints.

### 1. Initialize Session

* **Endpoint:** `POST /api/session/start`
* **Payload:**
```json
{
  "case_id": "RUN_2026_06A",
  "customer_id": "CUST_99103"
}

```


* **Behavior:** Queries `/opt/squeeze/shared/customer_history.db` to spin up an active conversational context for the specified customer. Returns a unique `session_token`.

### 2. Conversational Processing

* **Endpoint:** `POST /api/session/chat`
* **Payload:**
```json
{
  "session_token": "abc-123-xyz",
  "message": "My order never arrived. I demand an immediate refund to my card or I will escalate this to my legal counsel."
}

```


* **Behavior:** Appends the incoming token to the active chat context, logs the transaction, and returns the response payload generated by the backend routing logic.

### 3. State Commitment (The Lockpoint)

* **Endpoint:** `POST /api/session/action`
* **Payload:**
```json
{
  "session_token": "abc-123-xyz",
  "decision": "ESCALATE" 
}

```


* **Allowed Decisions:** `REIMBURSE`, `DENY`, `ESCALATE`
* **Behavior:** Formally terminates the session context, executes any structural changes to the mock gateway interface, and writes the definitive result to the evaluation output registry.

## 4. The Isolated Pipeline and Gate Evaluation Mechanics

When a workflow iteration initiates, execution routes through the state machine strictly separating the implementation space from the test suite.

### Context-Blocked Dispatch Loop

1. 
**The Implementer Agent** reads the authored policy spec (`spec-N.md`) from its home workspace. It designs the internal prompt matrices, classification heuristics, and call routing logic. It has execution access to `/api/session/chat` but cannot see the test definitions or negative evaluation vectors.


2. 
**The Exerciser Agent** reads the identical policy spec (`spec-N.md`) in complete isolation. It converts policy boundary requirements into automated, multi-turn dialogue trees. It is blind to the prompt configurations and system source files.



### Gate Sentinel Validation Criteria

When the evaluation phase triggers, `gate_sentinel.py` validates the artifacts against the machine gates:

* 
**Gate A (Editorial Check):** Asserts that the forward documentation has undergone explicit tracking adjustments and content modifications by the coordinating framework before code execution began.


* **Gate B (Machine Acceptance Squeeze):** Fires the Exerciser's scenario scripts through the REST API runtime app. It validates that:
1. **Positive Claims:** Genuinely valid customer refund scripts successfully trigger a `REIMBURSE` payload at `/api/session/action`.
2. 
**Negative Faults:** Adversarial inputs, social-engineering tracks, or prompt-injection attempts fail or fall back to an explicit `ESCALATE` or `DENY` payload at `/api/session/action`, precisely matching the fault location predicted by the Exerciser.


3. 
**Invariants:** The full index of files in `archive_ledger/` passes with zero regression deviations.




* 
**Gate C (Authored Authority Verification):** Maps every distinct clause enumerated by the upstream policy author directly to a passed execution test case in the validation metrics. If the policy dictates a specific restriction regarding multi-account checking, but the Exerciser's check matrix fails to explicitly validate this constraint against the API, Gate C aborts the pipeline.
