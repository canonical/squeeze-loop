# System Specification: The In-Band Deliverable Layer (Example C: API Contract Guard)

This specification outlines the structural, interface, and execution rules for the code artifacts built inside the `implementer` and `exerciser` environments. These components occupy the *in-band* deliverable space, bounded from above by the **API Governance Manifest (Upper Bound)** and from below by the **Live HTTP Server & Schema Linters (Lower Bound)**.

To maintain compliance with physical isolation barriers, these two execution bands operate under total context-blindness relative to each other. Their alignment is forced exclusively through automated cross-plane verification during gate evaluation.

```text
         [ UPPER BOUND: API Governance Manifest (API_POLICY_081) ]
                                     │
               ┌─────────────────────┴─────────────────────┐
               ▼                                           ▼
 ┌───────────────────────────┐               ┌───────────────────────────┐
 │     IMPLEMENTER BAND      │               │      EXERCISER BAND       │
 │   (main.py & openapi.json)│               │    (test_matrix.json)     │
 └───────────────────────────┘               └───────────────────────────┘
               │                                           │
               └─────────────────────┬─────────────────────┘
                                     ▼
         [ LOWER BOUND: Live Localhost Runtime + Static Schema Linter ]

```

## 1. The Implementer’s Band: Code & Schema Architecture

The `implementer` account is responsible for producing both the running web server application and its matching public OpenAPI contract file.

* **File Targets:** * `/home/implementer/src/main.py` (The API application logic, e.g., FastAPI)
* `/home/implementer/src/openapi.json` (The generated or manually updated contract schema)


* 
**Context Environment:** This workspace contains the raw text of the active policy clauses (`spec.txt`). It is physically blocked from viewing any test matrix or validation script generated by the exerciser.



### Execution Constraints

* **Decoupled Responses:** The server routes must explicitly translate database records into output schemas that scrub internal auto-incrementing database primary keys, mapping resources exclusively using universally unique identifiers (`user_uuid`) as mandated by the upper bound.
* **Strict Error Catching:** The application code must implement global exception handling middleware. It must capture unhandled parameter exceptions or database faults and format them into an explicit `400 Bad Request` JSON payload. It is strictly forbidden from leaking raw Python traceback text frames to the client.

The `exerciser` account produces the validation criteria. Because it operates completely blind to the implementation code within `main.py`, its deliverables represent a pure, unpolluted testing harness derived entirely from the schema definition and policy boundaries.

* **File Target:** `/home/exerciser/conformance/test_matrix.json`

## 3. The Squeeze Connector: Split-Plane Cross-Check Runner

The execution bridge is managed by the automated orchestrator component (`gate_sentinel.py`) running under the `sentinel` user account. It maps data across the isolated agent environments, ensuring correctness is the fixed point that satisfies both planes simultaneously.

For every entry defined in the exerciser's matrix, the runner executes the following evaluation loop:

```python
import json
import requests
import subprocess
import sys
import time

def run_split_planes_squeeze():
    # 1. Load the independent validation scenarios written by the Exerciser
    with open("/home/exerciser/conformance/test_matrix.json") as f:
        matrix = json.load(f)

    # 2. Boot the Implementer's live server process (Plane 2 Lower Bound)
    server_proc = subprocess.Popen(
        ["/opt/squeeze/runtime_app/run_server.sh"],
        stdout=subprocess.PIPE, stderr=subprocess.PIPE
    )
    time.sleep(2)  # Give the local network socket time to bind to localhost:8000

    try:
        for case in matrix["endpoints"]:
            url = f"http://127.0.0.1:8000{case['path']}"
            
            # Fire the programmatic HTTP request into the live runtime
            response = requests.request(
                method=case["method"],
                url=url,
                headers=case.get("headers", {}),
                json=case.get("payload", {})
            )

            # Enforce Gate B Execution Target
            if response.status_code != case["expected_status"]:
                print(f"GATE B FAILURE: Route {case['path']} returned wrong HTTP status code!")
                print(f"Expected: {case['expected_status']}, Got: {response.status_code}")
                sys.exit(1)

            # Verify response schema keys match the exact structural constraint
            response_json = response.json()
            if set(response_json.keys()) != set(case["expected_schema_keys"]):
                print(f"GATE B FAILURE: Document/Runtime Schema Mismatch on route {case['path']}!")
                print(f"Expected Keys: {case['expected_schema_keys']}, Got: {list(response_json.keys())}")
                sys.exit(1)

            # Verify strict absence of raw environment leak lines (Clause 3 Defense)
            if "forbidden_string_patterns" in case:
                response_text = response.text
                for pattern in case["forbidden_string_patterns"]:
                    if pattern in response_text:
                        print(f"GATE B FAILURE: Security Data Leakage! Found forbidden pattern '{pattern}' in output.")
                        sys.exit(1)

        print("GATE B SUCCESS: Split planes run in total in-band alignment.")

    finally:
        # Guarantee server cleanup across process terminations
        server_proc.terminate()

if __name__ == "__main__":
    run_split_planes_squeeze()

```

## 4. Automated Gate Coupling Focus

* 
**Gate B Clearance (No-Blend Verification):** The runner confirms that the implementer's runtime code does not drift from its public-facing declaration plane. If the code returns undocumented keys or throws unhandled framework errors when poked by the exerciser's matrix, the run fails.


* **Gate C Structural Check:** The sentinel extracts the `target_clauses` string arrays from the exerciser's test file. If any unique clause ID defined within the upper bound handbook (`api_policy_manifest.md`) lacks a corresponding targeted testing scenario block here, the pipeline aborts for coverage deficiency.
